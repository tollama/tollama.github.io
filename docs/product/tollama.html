tollama

Ollama for Time Series.
Run SOTA Time Series Foundation Models with a single API.

Time series foundation models are fragmented — incompatible libraries, conflicting dependencies, inconsistent APIs. tollama unifies 6 SOTA TSFMs into a single daemon with isolated runtimes, one REST API, and first-class agent integration.

// The Problem
Why another tool?

Every TSFM ships its own install, its own API, and its own dependency tree. Building on top of them means fighting fragmentation at every layer.

01 / FRAGMENTATION
Every model ships its own install.
Chronos needs torch, TimesFM needs JAX, Moirai needs einops. Installing multiple models in one environment leads to version hell.
02 / INCONSISTENT APIs
No common interface across models.
Different function signatures, parameter names, and output formats per family. Comparing models means writing adapters for each one.
03 / DEPENDENCY CONFLICTS
Models can't coexist.
torch 2.x and JAX fight over GPU memory allocation. Different CUDA requirements per family make unified environments impossible without isolation.
04 / AGENT FRICTION
LLMs can't call forecast models.
No standard tool interface for AI agents to discover, invoke, and chain time series forecasts. MCP, LangChain, A2A integrations must be built per model.

Quickstart

Install tollama, start the daemon, and run your first forecast in under a minute.

1) Install
python -m pip install tollama

# from source (dev):
python -m pip install -e ".[dev]"
2) Start the daemon
tollama serve
# check:
curl http://localhost:11435/api/version
tollama info --json
3) Pull + forecast
# pull a model
tollama pull chronos2

# direct forecast
curl -s http://localhost:11435/api/forecast \
  -H 'content-type: application/json' \
  -d @examples/chronos2_request.json

# progressive SSE stream
curl -N http://localhost:11435/api/forecast/progressive \
  -H 'content-type: application/json' \
  -d @examples/chronos2_request.json

Full walkthrough including runtime isolation, data ingest, SDK usage, and agent integrations: docs/how-to-run.md.

// Architecture
A daemon that
manages everything.

The tollamad daemon orchestrates isolated virtual environments per model family, communicating via JSON-lines stdio protocol.

  • tollamad Daemon managing model lifecycle and requests
  • runners/ Per-family isolated venvs under ~/.tollama/runtimes/
  • JSON-lines stdio protocol between daemon and runners
  • SSE Server-Sent Events for progressive streaming
  • Prometheus /metrics endpoint for observability
  • API-key Optional authentication + usage metering

// Advanced Features
Beyond basic forecasting.
🧠
Structured Intelligence

Combined analysis + forecast in a single call via /api/report. Returns narrative and quantitative results in one structured response.

📡
Progressive Forecasting

Real-time forecast refinement via SSE. Watch forecasts improve through stages via /api/forecast/progressive.

📥
Data Ingest

Upload CSV or Parquet files via data_url parameter or dedicated upload endpoints. No manual data formatting required.

🔗
Workflow Chaining

Multi-step orchestration via /api/pipeline. Chain forecasts, comparisons, what-if scenarios, and report generation.

🔮
What-if Scenarios

Explore alternative futures via /api/what-if, /api/counterfactual, and /api/scenario-tree.

🔒
Isolated Runtimes

Each model family gets its own venv under ~/.tollama/runtimes/. No dependency conflicts. Ever.

⚡
Auto-Forecast & Comparison

Run all pulled models against the same data with /api/auto-forecast. Automatic metric comparison and best-model selection.

🐳
Docker + Auth + Metrics

Containerized deployment with Docker Compose, optional API-key auth, usage metering at /api/usage, and Prometheus /metrics.
